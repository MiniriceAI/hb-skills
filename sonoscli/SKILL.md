# 核心定位

将知识工作者的自然语言需求转化为具体的 CLI 操作或工具调用，串联任务管理、日历、会议纪要和工作状态汇报，减少在不同工具间切换的摩擦。核心价值是：一句话输入，结构化输出，并给出下一步建议。

---

# 触发场景

- 用户说"帮我从这次会议记录里提取行动项"
- 用户说"生成今天的日报/周报/站会汇报"
- 用户说"把这个任务加到我的看板"
- 用户说"检查我今天有什么待办"
- 用户提供一段会议录音文字稿或文字笔记，需要整理

# 支持的集成

| 集成类别 | 工具示例 | 主要操作 |
|---------|---------|---------|
| 任务管理 | Notion、Linear、Jira、Trello、飞书多维表 | 创建/更新/查询任务 |
| 日历 | Google Calendar、飞书日历、Outlook | 查看/创建/修改日程 |
| 即时通讯 | Slack、飞书、钉钉 | 发送消息、获取频道历史 |
| 文档 | Notion、Google Docs、飞书文档 | 创建/更新文档 |
| 代码托管 | GitHub、GitLab | 查询 PR、Issue 状态 |
| 邮件 | Gmail、Outlook | 发送/搜索邮件 |

若用户未指定工具，默认询问："你用哪个工具管理任务/日历？"

# 工作流

## Step 1: 意图识别

分析用户输入，确定操作类型：

```
意图类型:
  A. 任务操作     → 创建/更新/查询/完成任务
  B. 日历操作     → 查看/创建/修改日程
  C. 会议纪要处理 → 提取行动项 / 整理要点 / 分配负责人
  D. 状态汇报     → 生成日报/周报/站会摘要
  E. 信息聚合     → 跨工具汇总当前待办、阻塞项
```

若输入模糊（如"帮我整理一下"），询问：
> "你是想整理任务列表、生成汇报，还是处理会议记录？"

## Step 2: 参数提取

根据意图类型，提取所需参数：

**任务创建参数：**
```
- 标题（必填）
- 负责人（可选，未提及则留空或询问）
- 截止日期（可选，"下周五"→解析为具体日期）
- 优先级（可选，默认 Medium）
- 所属项目/标签（可选）
- 描述（可选，从上下文生成）
```

**会议纪要参数：**
```
- 会议主题
- 参会人列表
- 会议时间
- 原始文字内容
```

**汇报参数：**
```
- 汇报类型（日报/周报/站会）
- 时间范围
- 数据来源（哪个看板/项目）
```

## Step 3: 自然语言到操作映射

**示例映射表：**

| 用户输入 | 识别意图 | 执行操作 |
|---------|---------|---------|
| "把'优化登录流程'加到本周待办，交给小李" | 任务创建 | 创建任务，负责人=小李，截止=本周日 |
| "今天下午有什么会议" | 日历查询 | 查询今天 12:00-23:59 的日程 |
| "帮我看看 Linear 上有哪些阻塞的任务" | 任务查询 | 查询 status=blocked 的所有任务 |
| "生成今天的站会汇报" | 状态汇报 | 按模板聚合昨日完成/今日计划/阻塞 |
| "把这段会议记录整理一下" | 会议纪要 | 提取行动项+决策+下次跟进 |

**自然语言时间解析规则：**
```
"今天"     → 当日日期
"明天"     → +1 天
"下周X"    → 下周对应星期，时间默认 EOD 18:00
"这周内"   → 本周五 EOD
"月底前"   → 当月最后一个工作日
"ASAP"     → 今日 EOD，优先级 High
"尽快"     → 同 ASAP
```

## Step 4: 会议纪要行动项提取

当用户提供会议文字内容时，执行以下结构化提取：

**提取框架：**
```
1. 扫描决策（寻找"我们决定"、"确认"、"同意"等关键词）
2. 扫描行动项（寻找"负责"、"会"、"要"、"去做"、"跟进"等动词）
3. 扫描问题/阻塞（寻找"但是"、"问题是"、"卡在"、"不确定"等）
4. 扫描下次会议安排
```

**行动项判断标准（必须满足其中 2 条）：**
- 有明确动词（做/完成/确认/输出/发送）
- 有隐含或明确的负责人
- 有时间压力或截止日期
- 与会议目标直接相关

**行动项输出格式：**
```
[ ] [负责人] [动作描述] — 截止：[日期] — 优先级：[H/M/L]
```

若负责人不明确，标注 `[待分配]`，询问用户。

## Step 5: 生成日报/周报/站会汇报

**站会（Daily Standup）模板：**
```
昨日完成：
  · [任务名] — [项目]
  · ...

今日计划：
  · [任务名] — 预计耗时：Xh
  · ...

阻塞/风险：
  · [问题描述] — 需要：[什么支持]
  （无阻塞则写：无）
```

**周报模板：**
```
本周完成（[日期范围]）：
  [主要成果1]
  [主要成果2]

关键数据：
  · [指标1]：[数值] (vs 上周：+X%)
  · [指标2]：[数值]

下周计划：
  · [任务/目标1]
  · [任务/目标2]

风险与需要支持：
  · [如有]
```

## Step 6: 确认并执行

在执行任何写操作（创建任务、发送消息、修改日历）前，输出操作预览：

```
即将执行以下操作，确认？(y/n)

> 创建任务：优化登录流程
  负责人：小李
  截止日期：2026-04-05（周日）
  项目：用户体验优化
  优先级：Medium
```

用户确认后执行，并输出结果链接（如有）。

## Step 7: 后续建议

每次操作完成后，给出 1-2 条相关建议：

```
完成！已创建任务 #1234。

建议后续操作：
→ 是否需要为这个任务设置检查提醒？（建议截止前 1 天）
→ 是否需要在 Slack 通知小李这个任务已分配？
```

# 输出格式

**任务创建确认：**
```
[已创建任务]
标题：[任务名]
负责人：[姓名]  截止：[日期]  优先级：[级别]
链接：[url（如有）]
```

**会议行动项提取：**
```
会议：[主题] | [日期] | 参会：[人员]

行动项（X条）
[ ] [负责人] [行动] — 截止：[日期]
...

关键决策
· [决策1]
· [决策2]

待解决问题
· [问题1] — 负责跟进：[人]

下次会议：[时间/待定]
```

**状态汇报：**
直接输出可复制粘贴的格式，不加额外解释。

# Gotchas

**意图误判**
- "整理"这个词歧义高——可能是整理任务、整理文档、整理会议记录，必须确认
- "发给XXX"可能是发任务也可能是发消息，需要明确渠道

**时间解析**
- "下周五"在周五当天说 = 下一个周五（+7天），不是今天
- "月底"如果当月只剩 2 个工作日，应提示用户是否合理
- 用户说"尽快"时不要解析成今天——应询问："截止今天，还是本周内？"

**行动项提取**
- 会议记录里"大家觉得应该..."不是行动项，是建议——需要有具体负责人才算
- "XX 会帮忙看看" = 软承诺，标注为低置信度行动项，提示用户确认
- 多人会议中，第三人称提及的人名不一定是与会者，核实前用 `[待确认：XX]`

**工具集成**
- 不同工具的字段映射不同（Linear 有 Priority，Notion 没有原生 Priority 字段），需要根据目标工具调整
- 若用户未登录/未授权目标工具，先提示授权，不要继续执行
- 批量操作（如一次创建 10 个任务）必须先展示列表让用户确认，不要直接执行

**汇报质量**
- 站会汇报不要写"正在进行中"——只写昨日完成和今日计划，进行中的任务属于今日计划
- 周报数据要有对比基准（vs 上周、vs 目标），没有基准的数字没有意义
- 不要在汇报里写"无问题"——若真无阻塞，写"本周无阻塞"更专业
